### 🔍 개념

- **애플리케이션 캐시**는 서버나 브라우저 외에 **애플리케이션 코드 내부에서 수행되는 캐싱 전략**
- 주로 **서버 코드 내 메모리**, 혹은 **클라이언트 코드(JavaScript 등)** 내부에 구현
- 불필요한 네트워크 요청, 반복 연산을 줄이고 성능 향상

---


### 🧠 서버 측 애플리케이션 캐시

- **자바, Node.js 등 서버 코드 내에서 구현**
- 프레임워크 혹은 라이브러리 사용 (예: Caffeine, Guava, Ehcache, LRU Cache 등)

### 예: Java (Caffeine)

- 최근 조회된 사용자 정보, 권한, 설정 등 자주 조회되는 데이터 캐싱
- TTL, 최대 용량, 자동 제거 정책 등 다양한 설정 가능

### 특징

- 캐시 로직을 애플리케이션에 명시적으로 작성
- 인메모리 기반 → 빠른 접근
- 서버 재시작 시 캐시 초기화됨 (휘발성)

---


### 🧠 클라이언트 측 애플리케이션 캐시

- **브라우저 내 JavaScript 코드에서 직접 구현**
- SPA의 상태 관리 라이브러리나 fetch 요청 결과를 저장

### 예:

- React Query, SWR, Apollo Client 등에서 요청 결과를 캐싱
- 동일 API 요청에 대해 재요청 없이 메모리에서 응답

---


### 🔄 캐시 전략

- **Write-through**: 저장 시 캐시와 DB 모두에 반영
- **Write-back**: 먼저 캐시에만 저장, 이후 비동기로 DB 반영
- **Read-through**: 캐시에 없으면 DB 조회 후 캐시에 저장
- **Cache-aside (Lazy Load)**: 수동으로 캐시에 저장 및 갱신

---


### 🔐 주의 사항

- 데이터 변경에 민감한 경우 캐시 사용 주의
- TTL 설정 또는 명시적 삭제 필요
- 메모리 한계 고려 → 캐시 크기/만료 정책 조정
- 분산 시스템에서는 개별 서버 캐시에 의존하면 **불일치 발생 가능**

---


### ✅ 주요 활용 사례

- 자주 바뀌지 않는 설정값, 코드 테이블
- 외부 API 응답 결과 캐싱
- 사용자 세션 상태나 임시 계산 결과 저장
- 단건 쿼리 결과 캐싱으로 성능 개선
