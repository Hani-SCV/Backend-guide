### 개념


> - DB 캐시는 데이터베이스 쿼리 결과를 메모리에 저장해 **읽기 성능을 향상**시키는 구조  
>   
> - **DB 자체 기능**, 또는 외부 캐시 계층(Redis 등)과 결합하여 구현  
>   
> - 읽기 부하가 높은 시스템에서 병목 최소화


### 1. DB 내장 캐시 (Built-in)

- RDBMS가 자체적으로 제공하는 쿼리 결과 캐시, 인덱스 캐시
- 예: Oracle의 **Result Cache**, MySQL의 **Query Cache**(현재는 비활성화됨)

### 2. 외부 캐시 연동 (2계층 캐시)

- 애플리케이션이 DB 호출 전 **메모리 캐시(Redis, Memcached 등)** 조회
- **Cache Miss 시 DB 조회 → 캐시에 결과 저장**
- 성능과 유연성은 좋지만 **동기화 이슈** 고려 필요

### 캐시 갱신 전략

- **TTL(Time To Live)** 설정 → 자동 만료
- **수동 갱신(Invalidate or Refresh)** → 업데이트/삭제 시 캐시 무효화
- **Event 기반 갱신** → DB 변경 이벤트를 수신해 캐시 갱신 (CDC, 메시지 큐 등 활용)

### 패턴: Cache Aside (Lazy Load)

1. 애플리케이션은 먼저 캐시에서 데이터 조회
2. 없다면 DB에서 가져온 후 캐시에 저장
3. 이후 동일 요청은 캐시에서 처리

→ 가장 일반적이며 유연성 높음


### 데이터 정합성 고려

- **읽기 성능 향상**에 초점 → 쓰기 또는 갱신이 빈번한 경우엔 비추천
- **쓰기 작업 후 캐시 무효화 또는 갱신 로직 필수**
- **강한 정합성**이 필요한 경우 캐시보다 직접 DB 접근이 나음

### 활용 예시

- 자주 조회되는 코드 테이블, 설정값
- 뉴스 목록, 인기 게시물 목록 등 변동이 적은 쿼리
- 통계 데이터, 분석 리포트 캐싱

### 주의할 점

- 캐시와 DB 간 데이터 불일치 가능성
- TTL 지나기 전 데이터가 변경될 경우 → **Stale Data 문제**
- 캐시 과잉 사용은 **복잡도 증가** 및 **메모리 낭비**
