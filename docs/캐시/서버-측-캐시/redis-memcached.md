### 개념


> - **서버 측 캐시**는 DB 조회나 연산 비용이 높은 데이터를 메모리에 저장해 성능을 높이는 방식  
>   
> - 대표적인 인메모리 캐시 저장소: **Redis**, **Memcached**  
>   
> - 둘 다 **Key-Value 기반 저장소**이며, 빠른 읽기/쓰기를 위해 메모리를 사용


### 캐시 사용 목적

- DB 부하 감소
- 응답 속도 향상
- 동일 요청에 대한 반복 계산 제거 (ex. 검색 결과, 세션, 권한 정보 등)

---


### Redis 특징

- **데이터 구조 지원**: 문자열, 리스트, 해시, 셋, 정렬셋 등
- **지속성(Persistence)**: 디스크 저장 가능 (AOF, RDB 방식)
- **Pub/Sub**, Lua 스크립트, 트랜잭션 지원
- TTL(Time-To-Live) 설정 가능 → 만료 자동 관리
- 세션 저장소, 캐시, 메시지 브로커 등 다양한 용도로 활용

### Memcached 특징

- 오직 **Key-Value 문자열 저장**만 가능 (구조 없음)
- **단순하고 빠른 캐시 전용** 시스템
- TTL 설정 지원
- Persistence(지속성) 없음 → 휘발성 캐시 용도에 적합
- 복잡한 기능 없이 읽기 성능이 매우 우수

### Redis vs Memcached 비교


| 항목     | Redis                   | Memcached      |
| ------ | ----------------------- | -------------- |
| 데이터 구조 | 다양한 자료형 지원              | 문자열만 가능        |
| 지속성    | AOF, RDB                | 지원하지 않음        |
| 멀티 스레드 | 단일 스레드 (v6부터 일부 지원)     | 멀티 스레드         |
| 사용 용도  | 세션, 캐시, 큐, Pub/Sub 등 다양 | 단순 캐싱          |
| 확장성    | 클러스터링 지원                | 샤딩 기반 수동 분산 필요 |
| 보안     | 인증, TLS 지원              | 보안 기능 거의 없음    |


### 동작 흐름 예시

1. 애플리케이션 요청 → 캐시에서 키 조회
2. 캐시에 없으면 DB에서 조회 → 결과를 캐시에 저장
3. 이후 동일 요청은 캐시에서 바로 응답
4. TTL 지나면 자동 제거되거나, 수동으로 갱신/삭제

### 주의할 점

- **캐시 무효화(Cache Invalidation)** 전략 필요: 갱신 시점/조건 명확히 설정
- **일관성 보장**이 중요한 데이터는 캐시 사용에 신중
- TTL 설정 없이 방치하면 **오래된 데이터 제공 가능성**
- 캐시가 DB보다 최신 데이터가 아닐 수 있음

### 언제 어떤 걸 써야 할까?

- **Redis**: 기능이 많고 복잡한 데이터 처리, 세션/토큰 저장, 지속성이 필요할 때
- **Memcached**: 단순히 읽기 성능을 높이고 싶을 때, 리소스가 제한될 때
