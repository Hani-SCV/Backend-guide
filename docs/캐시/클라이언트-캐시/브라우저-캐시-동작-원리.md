### 개념


> - 브라우저는 서버에서 받은 리소스(HTML, CSS, JS, 이미지 등)를 **로컬에 저장**하여 재요청 없이 빠르게 로드  
>   
> - HTTP 헤더(`Cache-Control`, `ETag`, `Last-Modified` 등)를 기반으로 **캐시 여부와 유효성 판단**


### 동작 흐름

1. 클라이언트 → 서버: 최초 요청 (예: GET /style.css)
2. 서버 → 클라이언트:

    • 응답 헤더에 캐시 지시어 포함


    `Cache-Control: max-age=3600`


    `ETag: "abc123"`

3. 클라이언트: 리소스를 캐시에 저장
4. 이후 동일 요청 시, 브라우저는 먼저 **캐시 검사**
5. 조건부 요청 발생:

    • `If-None-Match: "abc123"`


    → 서버는 변경 없을 경우 `304 Not Modified` 반환


    → 변경 있으면 새 리소스 전달


### 주요 헤더

- `Cache-Control`: 캐시 지속 시간, 공개/비공개 설정
- `ETag`: 리소스 식별용 고유 값
- `Last-Modified`: 최종 수정 시간
- `Expires`: 캐시 만료 시점 (HTTP/1.0)
- `Pragma`: 과거 브라우저용 캐시 제어

### 장점

- **속도 향상**: 리소스 재요청 방지
- **트래픽 절감**: 서버 부하 감소
- **사용자 경험 개선**: 로딩 시간 단축

### 단점

- **오래된 리소스 사용 가능성** (만료 정책 잘못 설정 시)
- **캐시 무효화 어려움**: 빌드 시 파일명 해시 처리 필요
- **디버깅 시 혼란**: 캐시된 데이터로 인해 결과 지연 확인 어려움

### 보안 고려사항

- 민감 정보는 캐시 금지 → `Cache-Control: no-store`
- 인증 요청에는 `no-cache`, `private` 설정 권장
- HTTPS 환경에서만 캐시 정책 엄격 적용
- 보안상 중요한 데이터는 절대 브라우저 캐시에 저장하지 말 것

### 무효화 전략

- **파일명에 해시값 추가 (예: app.ab12cd.js)**
- **빌드 자동화 도구로 해시 리네이밍 적용 (Webpack 등)**
- **ETag/Last-Modified 기반 조건부 요청 활용**
- 캐시 만료시간 조절로 유연한 업데이트 반영

### Tip

- 변경 잦은 리소스는 `no-cache` 또는 짧은 `max-age` 설정
- 정적 자산에는 `Cache-Control: public, max-age=31536000, immutable` 권장
- 디버깅 시 브라우저 개발자 도구에서 캐시 비활성화
- CDN 및 리버스 프록시 캐시 정책도 일관성 있게 관리
