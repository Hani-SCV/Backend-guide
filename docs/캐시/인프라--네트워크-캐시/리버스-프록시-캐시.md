### 개념


> 리버스 프록시는 클라이언트와 서버 사이에 위치하여 요청을 대신 처리하는 서버다. **리버스 프록시 캐시**는 이 프록시 서버가 응답 데이터를 캐싱해 두고, 동일 요청이 들어오면 원서버가 아닌 프록시에서 응답을 제공하는 방식이다.  
> 대표적인 구현체: **Nginx**, **Apache**, **Varnish**


### 동작 흐름

1. 클라이언트가 요청을 보냄 → 리버스 프록시가 수신
2. 캐시에 동일 리소스가 있으면 바로 응답 (Cache Hit)
3. 없으면 원서버에 전달 → 응답을 받아 캐시에 저장 후 클라이언트에 전달 (Cache Miss)
4. 이후 동일 요청은 캐시된 응답을 사용

### 캐시 구성 요소

- **캐시 저장소**

    메모리 또는 디스크 기반 저장 (Varnish: 메모리 / Nginx: 디스크)

- **TTL 설정**

    `Cache-Control`, `Expires` 등의 응답 헤더 기준으로 캐시 지속 시간 설정

- **키 구성**

    요청 URI + 헤더 값 등으로 구성하여 캐시 범위 조정 가능


### 장점

- 동적 요청도 캐시 가능
- 서버 부하 감소 및 응답 속도 향상
- SSL 종료 처리와 같이 부가 기능 수행 가능
- 서버 장애 시 캐시된 콘텐츠로 응답 가능

### 단점 및 주의사항

- 캐시 무효화 전략 필요 (API 캐싱 시 중요)
- 로그인 등 사용자 맞춤형 데이터 캐싱 주의
- 메모리 또는 디스크 용량 제한 고려

### 보안 고려사항

- 인증 정보 캐싱 방지
- 쿠키, 토큰 포함된 응답은 `Cache-Control: private` 설정
- HTTPS 트래픽 암복호화 처리 시 리소스 부담 고려

### 캐시 제어 전략

- **정적 리소스**: `public, max-age=31536000`
- **동적 콘텐츠**: `Cache-Control: s-maxage=60, stale-while-revalidate=30`
- **API 응답**: 로그인 여부 판단 후 조건부 캐시 적용
