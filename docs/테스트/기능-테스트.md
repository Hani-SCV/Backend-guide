### 개념


> - 사용자의 **기능 단위 동작 흐름을 테스트**  
>   
> - 단위 테스트보다 **상위 계층(컨트롤러 + 서비스 등)**을 포함  
>   
> - HTTP 요청/응답 중심의 테스트가 많음


### 목적

- 기능 흐름이 **요구사항대로 동작**하는지 확인
- **비즈니스 플로우** 검증 (예: 게시글 작성 → 조회 → 수정 → 삭제)
- 실제와 유사한 방식으로 **엔드포인트를 테스트**

### Java / Spring 실무 기준


| 기술                 | 설명                              |
| ------------------ | ------------------------------- |
| `@SpringBootTest`  | 전체 애플리케이션 컨텍스트 로드 후 통합 테스트      |
| `@WebMvcTest`      | 컨트롤러 슬라이스 테스트 (의존 서비스는 Mock 처리) |
| `MockMvc`          | HTTP 요청/응답 시뮬레이션                |
| `TestRestTemplate` | 실제 HTTP 호출 기반 테스트 (통합 테스트에서 사용) |
| `@Transactional`   | 테스트 종료 후 롤백 보장 (DB 변경 안 남김)     |


### 기능 테스트 예시 (Spring + MockMvc)


```java
@WebMvcTest(controllers = PostController.class)
class PostControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private PostService postService;

    @Test
    void 게시글_작성_성공() throws Exception {
        PostRequest request = new PostRequest("제목", "내용");

        mockMvc.perform(post("/posts")
                .contentType(MediaType.APPLICATION_JSON)
                .content(new ObjectMapper().writeValueAsString(request)))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.title").value("제목"));
    }
}
```


### 실무 팁

- **단위 테스트 + 기능 테스트 병행**이 기본
- 모든 API 엔드포인트에 대해 **기능 테스트 작성 권장**
- MockMvc는 컨트롤러 계층만 검증 → **서비스 계층 Mock 처리 필수**
- 실제 DB 사용 시 `@Transactional` 붙여야 테스트 간 영향 없음
- 테스트용 DB, S3, Redis, Kafka 등은 **로컬 테스트 환경 구성** 고려
