### 🧩 개념

- *하나의 단위(클래스, 함수, 메서드)**를 독립적으로 테스트
- 외부 의존성 없이 **내부 로직이 의도대로 동작하는지 검증**

---


### 🔍 목적

- 작은 단위의 기능을 빠르게 테스트해 **버그 조기 발견**
- 코드 리팩터링 시 **안정성 확보**
- TDD(Test Driven Development) 기반 테스트 작성에 핵심

---


### ✅ 단위 테스트 대상 예시

- 서비스 계층 비즈니스 로직
- 유틸리티 함수 (문자열 처리, 날짜 계산 등)
- 예외 처리, 조건 분기
- Validator, Converter 등 독립성 높은 컴포넌트

---


### 🛠 Java / Spring 실무 기준


| 기술                           | 설명                             |
| ---------------------------- | ------------------------------ |
| `JUnit5`                     | 테스트 프레임워크 표준                   |
| `Mockito`                    | 의존 객체(Mock)를 생성해 단위 테스트 수행     |
| `AssertJ`                    | 가독성 높은 테스트 검증 라이브러리            |
| `@Mock`, `@InjectMocks`      | 서비스 단위의 의존성 주입 및 Mock 처리       |
| `@WebMvcTest`, `@JsonTest` 등 | 특정 슬라이스만 테스트 가능 (컨트롤러, JSON 등) |


---


### 🧪 단위 테스트 예시 (Service 레이어)


```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private OrderRepository orderRepository;

    @InjectMocks
    private OrderService orderService;

    @Test
    void 주문_생성_정상_동작() {
        Order order = new Order("item", 1000);
        when(orderRepository.save(any())).thenReturn(order);

        Order result = orderService.createOrder("item", 1000);

        assertThat(result.getItem()).isEqualTo("item");
    }
}
```


---


### 🔧 실무 팁

- **Mock으로 외부 의존성 제거** → 테스트 속도 향상 & 안정성 확보
- *단일 책임 원칙(SRP)**에 따라 테스트 대상도 작게 분리
- *경계 테스트(Edge Case)**도 포함: 예외, 빈 값, 최대값 등
- 테스트명은 한글로 작성해도 좋음 (가독성 ↑)
- 테스트 커버리지 도구(JaCoCo 등)로 **테스트 누락 구간 점검**
