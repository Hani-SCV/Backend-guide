### 🧩 개념

- **여러 모듈 또는 계층이 함께 동작하는지 검증**하는 테스트
- 예: 컨트롤러 → 서비스 → 리포지토리 → DB까지 실제 흐름을 따라가며 테스트

---


### 🔍 목적

- 컴포넌트 간 **연동 문제** 조기 발견
- 실제 **애플리케이션 환경과 유사한 조건**에서 동작 검증
- 단위 테스트로는 놓칠 수 있는 **의존성 문제, 설정 오류** 등 포착

---


### ✅ 통합 테스트 대상 예시

- DB 연동 쿼리 흐름
- 외부 API 연동
- 트랜잭션 처리 확인
- 파일 업로드·다운로드 흐름
- 실제 HTTP 요청 흐름 (Spring에서는 `@WebMvcTest`, `@SpringBootTest` 활용)

---


### 🛠 Java / Spring 실무 기준


| 테스트 유형                     | 설명                            |
| -------------------------- | ----------------------------- |
| `@SpringBootTest`          | Spring 전체 컨텍스트 로드 후 통합 테스트 수행 |
| `@Transactional`           | 테스트 후 롤백 처리로 DB 초기화           |
| `@Testcontainers`          | 도커로 실제 DB 컨테이너 띄워 테스트 가능      |
| `MockMvc`, `WebTestClient` | HTTP 요청/응답 흐름 테스트 (서버 없이)     |


---


### 🔧 실무 팁

- 테스트 간 **데이터 격리** 철저히 (Rollback or Embedded DB)
- **Test Fixture** 분리 관리 (예: TestDataBuilder, 테스트 전용 SQL 스크립트 등)
- 테스트 실패 시 **로그·에러 메시지 상세하게** 확인할 수 있도록 구성
- 외부 API 연동은 실제 요청 대신 **MockServer**, `WireMock`, `MockWebServer` 등 사용
