### 🧠 배포 전략이란?

- **어플리케이션을 서비스 중단 없이 또는 최소화하며 변경하는 방법**
- 트래픽을 어떻게 분산하고, 어떻게 롤백할지까지 포함
- **CI/CD 파이프라인**에서 배포 단계에 해당

---


### 🔹 1. 단순 배포 (Recreate Deployment)

- 기존 버전을 중단하고 새 버전으로 교체
- **다운타임 발생 가능**
- 간단하지만 **무중단 요구 시 부적합**

🧾 예시:


```shell
docker stop old-app
docker run new-app
```


---


### 🔹 2. 롤링 배포 (Rolling Deployment)

- 서버를 하나씩 순차적으로 새 버전으로 교체
- **무중단 배포 가능**
- 롤백은 까다로울 수 있음

🧾 Kubernetes 예시:


```yaml
strategy:
  type: RollingUpdate
```


---


### 🔹 3. 블루-그린 배포 (Blue-Green Deployment)

- **Blue(기존), Green(신규)** 환경을 두고 전환
- 트래픽 스위칭만으로 빠르게 롤백 가능
- 인프라 리소스 2배 필요

🧾 Nginx 예시:


```plain text
location / {
  proxy_pass http://green-app; # 기존엔 blue-app
}
```


---


### 🔹 4. 카나리 배포 (Canary Deployment)

- 신규 버전을 **일부 유저에게만 점진적으로 배포**
- 문제 발생 시 빠르게 롤백 가능
- **A/B 테스트와 유사**

🧾 예시:

- 10% 트래픽만 신규 버전 처리 → 이상 없으면 점진 확대

---


### 🔹 5. 셰도우 배포 (Shadow Deployment)

- 실제 유저 트래픽을 **복사**해 새 버전에서 테스트
- 유저에겐 영향 없음
- **비용과 복잡도 높음**

---


### ✍️ 선택 기준


| 전략       | 무중단 | 롤백 용이 | 리소스 비용 | 사용 시기            |
| -------- | --- | ----- | ------ | ---------------- |
| 단순 배포    | ✖   | 보통    | 낮음     | 개발/테스트 환경        |
| 롤링 배포    | ⭕   | 어려움   | 중간     | 일반적인 서비스 운영      |
| 블루-그린 배포 | ⭕   | 쉬움    | 높음     | 중요 서비스, 빠른 전환 필요 |
| 카나리 배포   | ⭕   | 쉬움    | 중간     | 점진적 테스트 필요 시     |
| 셰도우 배포   | ⭕   | 무의미   | 높음     | 실제 부하 테스트 필요 시   |
