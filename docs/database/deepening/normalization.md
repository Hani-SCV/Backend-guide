# 정규화 & 비정규화

## 정규화 (Normalization)

> **데이터 중복 제거 + 무결성 보장**을 위한 구조 최적화 과정  
> 목적: 중복 최소화, **업데이트 이상 방지**, 일관성 유지

### 📌 대표적인 이상 현상

| 종류 | 설명 |
| --- | --- |
| 삽입 이상 | 일부 데이터 없으면 추가 불가 (ex. 고객 없는 주문 불가) |
| 삭제 이상 | 불필요한 삭제로 유의미한 데이터도 같이 삭제됨 |
| 갱신 이상 | 중복된 값 갱신 누락으로 불일치 발생 |

### 🧱 정규화 단계

| 단계 | 핵심 조건 |
| --- | --- |
| 1NF | 모든 컬럼 값이 **원자값**이어야 함 (컬럼에 여러 값 X) |
| 2NF | 1NF + **부분 함수 종속 제거** (기본키 전체에 종속되도록) |
| 3NF | 2NF + **이행적 함수 종속 제거** |
| BCNF | 3NF + 모든 결정자는 **후보키** |

> 실무는 대부분 **3NF까지만 적용**, 필요 시 BCNF 검토

---

## ⚠️ 비정규화 (Denormalization)

> **조회 성능 향상**을 위해 의도적으로 중복 허용 또는 테이블 통합  
> 목적: **JOIN 최소화**, 빠른 쿼리 응답

### ❌ 단점

- **데이터 중복 → 일관성 유지 어려움**
- 스키마 관리 복잡도 증가

### 🧪 실무 적용 예

- 조회 전용 뷰 or 캐시 테이블
- 고객 정보와 주문 내역을 한 테이블에 통합

---

## ⚖️ 정규화 vs 비정규화

| 항목 | 정규화 | 비정규화 |
| --- | --- | --- |
| 목적 | 데이터 무결성, 중복 제거 | 조회 성능 개선 |
| 장점 | 관리 용이, 이상 현상 제거 | 빠른 조회, JOIN 감소 |
| 단점 | JOIN 비용 증가 | 중복 관리 필요 |
| 활용 | OLTP (트랜잭션 중심 시스템) | OLAP, 리포트 시스템, NoSQL 등 |

---

### 🌐 실무 고려 사항

- 데이터 처리 목적에 따라 **정규/비정규 혼합 사용**
- **분산 DB / NoSQL 환경**은 비정규화 중심 설계 많음 (스키마 유연성 + 성능 위주)
