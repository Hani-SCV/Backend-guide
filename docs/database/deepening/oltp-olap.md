# 스키마 설계 전략 (OLTP vs OLAP)

### 🔹 OLTP (Online Transaction Processing)

- 목표: 빠르고 안정적인 트랜잭션 처리
- 특징
    - 실시간 CRUD 중심(은행, 주문, 로그인 등)
    - 정규화된 스키마 구조 → 중복 최소화
    - 짧은 트랜잭션, 응답 속도 최우선
- 설계 포인트
    - *1NF~3NF* 적용
    - 외래키·인덱스 적절 배치
    - 트랜잭션 래티시 최소화
    - 파티셔닝·샤딩으로 수평 확장 대비
- 예시: 인터넷 쇼핑몰 주문·결제, 온라인 뱅킹

### 🔹 OLAP (Online Analytical Processing)

- 목표: 대규모 데이터 분석·리포팅
- 특징
    - 집계·추세 분석, 복잡한 다차원 쿼리
    - 비정규화된 스키마(Star, Snowflake)
    - 대량 배치 처리(ETL) 최적화
- 설계 포인트
    - 팩트 테이블과 디멘션 테이블 구분
    - 스타 스키마: 조회 단순화, 조인 최소화
    - 스노우플레이크 스키마: 디멘션 정규화로 관리 효율
    - 인덱스·파티셔닝(범위 기준) 적용
    - 증분 로딩, Slowly Changing Dimension 고려
- 예시: BI 리포트, 대시보드, 데이터 웨어하우스

### ⚖️ 비교 요약

| 구분 | OLTP | OLAP |
| --- | --- | --- |
| 목적 | 실시간 트랜잭션 처리 | 대용량 데이터 분석·리포팅 |
| 스키마 | 정규화 (중복 최소화) | 비정규화 (스타/스노우플레이크) |
| 쿼리 특성 | 단일 행 단위 CRUD | 다차원 집계, 복잡한 조인 |
| 응답 요구사항 | 짧은 지연 (수 ms~수십 ms) | 대용량 처리, 배치 응답 허용 |
| 확장 방식 | 샤딩·파티셔닝으로 수평 확장 | MPP(Massively Parallel Processing), 파티셔닝 적용 |
| 데이터 업데이트 | 실시간 | 주기적(일·시간 단위 ETL) |

### 🔄 Hybrid & HTAP

- OLTP·OLAP 경계 흐려지는 HTAP(실시간 트랜잭션+분석) 등장
- 설계 시 실시간 분석·트랜잭션 성능 균형 고려

### 🎯 Tip

- OLTP는 정규화+인덱스 튜닝, OLAP는 비정규화+배치 최적화
- DW 로딩 시 변형 규칙(ETL) 문서화
- Slowly Changing Dimension Type1/2 전략 사전 결정
- 각 시스템에 맞는 모니터링 지표(CPU·IO·쿼리 레이턴시) 설정
