# N + 1

### N+1 문제란?

> 초기 **1개의 SELECT 쿼리**로 N개의 엔티티 조회  
> 이후 각 엔티티의 연관 데이터를 **N번 개별 조회** → 총 N+1 쿼리 발생  
> **Lazy Loading**에서 자주 발생  
> 📉 성능 저하의 대표 사례

### ⚠️ 발생 원인

- **지연 로딩(Lazy Loading)**: 연관 객체를 실제 접근 시 조회
- **프록시 객체**를 통해 DB 접근 시점까지 로딩 지연
- 연관 필드를 루프 돌며 접근 시 → **N개의 쿼리 반복**

### 🔧 해결 방법

| 방법 | 설명 |
| --- | --- |
| **즉시 로딩 (Eager Loading)** | JOIN으로 한 번에 조회 (`fetch = EAGER`) |
| **배치 사이즈 설정** | `IN` 조건으로 묶어 조회 (Hibernate `@BatchSize`) |
| **EntityGraph 사용** | JPA에서 연관 조회를 명시적으로 지정 |
| **JPQL/쿼리 튜닝** | `JOIN FETCH`, `IN`, `EXISTS` 등 사용 |
| **쿼리 분석 도구 활용** | Hibernate SQL 로그, Spring Actuator, DB 프로파일러 등 |

### 💡 Tip

- 대체로 EAGER는 위험함 → 직접 **`JOIN FETCH` 사용 권장**
- Spring Data JPA에서는 `@EntityGraph`로 안전하게 지정
- 반복문 내 연관 필드 접근 시 항상 N+1 의심
