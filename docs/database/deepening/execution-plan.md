# 실행 계획과 쿼리 최적화

## 실행 계획 (Execution Plan)

> 쿼리 수행 시 **DB 엔진이 선택한 실행 절차**  
> 어떤 테이블을 어떤 방식으로 읽고, 어떤 인덱스를 쓰는지 명시

```sql
EXPLAIN SELECT ...
-- 또는 MySQL: EXPLAIN ANALYZE, Oracle: EXPLAIN PLAN FOR ...
```

### 자주 등장하는 키워드

| 항목 | 설명 |
| --- | --- |
| `TABLE ACCESS FULL` | 인덱스 없이 전체 테이블 탐색 |
| `INDEX RANGE SCAN` | 인덱스 일부 범위 탐색 |
| `INDEX FULL SCAN` | 전체 인덱스 탐색 |
| `NESTED LOOP` | 루프 기반 조인, 소규모 데이터에 적합 |
| `HASH JOIN` | 대량 데이터 조인, 인덱스 없어도 가능 |
| `COST`, `ROWS` | 실행 예상 비용과 처리 예상 행 수 (낮을수록 좋음) |

> 실행 계획은 예측이며, 실제 실행 시 상황 따라 다를 수 있음

## 🤖 옵티마이저 (Optimizer)

- **쿼리 최적 실행 계획을 선택**하는 DB 엔진
- **종류**
    - **비용 기반(CBO)**: 통계 기반 비용 평가 → 최신 DB 대부분 사용
    - **규칙 기반(RBO)**: 고정 규칙 우선순위

## 🛠️ 쿼리 최적화 전략

1. **인덱스 활용**
    - WHERE 조건절에 맞는 인덱스 유무 확인
    - 정렬/조인 컬럼도 인덱스 고려
2. **필요한 컬럼만 SELECT**
    - `SELECT *` 지양 → IO 비용 증가
3. **불필요한 DISTINCT, GROUP BY 제거**
    - 데이터량 많을수록 성능 저하 심각
4. **서브쿼리 대신 JOIN 고려**
    - 단순한 서브쿼리는 `JOIN`이 성능 우수
5. **페이징 주의**
    - `OFFSET` 클수록 느림 → 커서 방식 또는 **Seek 방식** 고려
6. **통계 정보 최신화**
    - `ANALYZE`, `UPDATE STATISTICS` 등 → 옵티마이저 정확도 향상

### 🔍 튜닝 실무 팁

- **실행 계획 먼저 확인 → 병목 지점 파악**
- 인덱스가 있더라도 옵티마이저가 사용하지 않을 수 있음
- **복합 인덱스** 사용 시 **선두 컬럼 조건 확인 필수**
- 인덱스 힌트(`USE INDEX`, `INDEX HINT`)는 **최후 수단**
