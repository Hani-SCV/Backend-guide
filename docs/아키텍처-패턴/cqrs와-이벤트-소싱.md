### 🧩 CQRS (Command Query Responsibility Segregation)

- 명령(Write)과 조회(Read)를 분리해 각각 최적화하는 아키텍처 패턴
- 쓰기 모델은 상태 변경과 비즈니스 로직 처리
- 읽기 모델은 빠른 데이터 조회에 집중
- 각 모델을 독립적으로 확장 가능

---


### 📜 이벤트 소싱 (Event Sourcing)

- 현재 상태를 저장하지 않고, 상태 변경 이벤트를 순차적으로 저장
- 시스템 상태는 이벤트를 재생해 복원
- 모든 변경 이력을 기록해 감사 및 디버깅에 유리

---


### ⚡ 장점

- CQRS: 읽기/쓰기 분리로 성능과 확장성 향상
- 이벤트 소싱: 완전한 변경 기록과 복구 가능
- 비동기 이벤트 처리로 복잡한 비즈니스 로직 구현에 유리

---


### ⚠️ 단점

- 설계 복잡성 증가
- 이벤트 버전 관리 및 저장소 설계 필요
- 디버깅과 테스트 어려움
- 높은 학습 곡선

---


### 🛠 활용 사례

- 복잡한 도메인 모델, 강력한 감사 추적 요구 시스템
- 금융, 예약, 주문 처리 시스템 등
