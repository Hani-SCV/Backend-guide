### 🧠 개념

- 애플리케이션을 **작고 독립적인 서비스 단위로 분할**한 구조
- 각 서비스는 독립적으로 개발, 배포, 확장 가능
- 서비스 간 통신은 주로 HTTP/REST, 메시징 큐 등으로 이뤄짐

---


### 특징

- 각 서비스는 독립적인 **비즈니스 도메인**을 책임
- 기술 스택, 데이터베이스도 서비스별로 다를 수 있음
- 분산 시스템이므로 네트워크 지연, 장애 처리 고려 필요

---


### ✅ 장점

- **유연한 확장성**: 서비스별로 필요에 따라 독립 확장 가능
- **개발 속도 향상**: 팀별로 서비스 분리하여 병렬 개발 가능
- **배포 독립성**: 한 서비스 수정 시 전체 재배포 불필요
- **장애 격리**: 한 서비스 장애가 전체 서비스에 영향 최소화

---


### ⚠️ 단점

- **복잡한 운영**: 서비스 간 통신, 모니터링, 로깅 필요
- **데이터 일관성 문제**: 분산 트랜잭션 관리 어려움
- **초기 구축 비용**: 인프라 자동화, 서비스 디스커버리 등 필요
- **네트워크 부하** 증가 및 지연 문제

---


### 🧩 핵심 구성 요소

- API Gateway
- 서비스 디스커버리
- 분산 트레이싱
- 메시징 시스템
- 컨테이너 오케스트레이션 (예: Kubernetes)

---


### ✍️ 적합한 상황

- 대규모 시스템, 빠른 확장과 유연한 배포 필요할 때
- 독립적 팀이 여러 서비스 병렬 개발 시
- 다양한 기술 스택 혼용 및 복잡한 비즈니스 로직 존재 시
