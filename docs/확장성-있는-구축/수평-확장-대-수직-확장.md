### 🔍 개념

- **확장성(Scalability)**: 시스템이 트래픽/데이터 증가에도 **성능 저하 없이 대응** 가능한 능력
- **수직 확장(Vertical Scaling)** ⬆️: 서버 성능(CPU, RAM, 디스크) 강화
- **수평 확장(Horizontal Scaling)** ➡️: 서버/서비스 인스턴스 추가 → 부하 분산

---


### ⚙️ 비교


| 항목    | 수직 확장 ⬆️              | 수평 확장 ➡️                   |
| ----- | --------------------- | -------------------------- |
| 방법    | 서버 업그레이드 (CPU, RAM 등) | 서버/서비스 추가                  |
| 장점    | 구현 간단, 기존 구조 유지       | 거의 무한 확장 가능, 장애 시 다른 서버 처리 |
| 단점    | 한계 존재, 비용 증가          | 세션 관리/데이터 동기화 필요, 구조 복잡    |
| 사용 사례 | 소규모 서비스, DB 서버        | 대규모 서비스, 웹/애플리케이션 서버       |


---


### ✅ 장점

- **수직 확장**
    - 구현이 쉽고 구조 변경 최소
    - 기존 애플리케이션 그대로 사용 가능
- **수평 확장**
    - 트래픽 급증 대응 용이
    - 장애 시 다른 서버가 처리 가능
    - 클라우드 환경에 최적화

---


### ⚠️ 단점 / 주의점

- **수직 확장**
    - 물리적 한계 존재
    - 비용 증가 → 성능 대비 효율 떨어짐
- **수평 확장**
    - 세션 공유/DB 동기화 필요
    - 아키텍처 복잡도 증가
    - 로드 밸런서 및 분산 캐시 필요

---


### 🛠 구현 예시


### 수평 확장 + 로드 밸런서 (Nginx)


```plain text
upstream app_servers {
    server 192.168.0.101:3000;
    server 192.168.0.102:3000;
    server 192.168.0.103:3000;
}

server {
    listen 80;
    location / {
        proxy_pass http://app_servers;
    }
}
```


### 수직 확장 (개념)


```plain text
기존 서버: 4CPU, 8GB RAM → 업그레이드 → 8CPU, 32GB RAM
```


---


### 💡 활용 사례

- 수직 확장: 소규모 DB 서버, 테스트/개발 환경 ⬆️
- 수평 확장: 웹/애플리케이션 서버, 클라우드 기반 서비스 ➡️
- 혼합 전략: 핵심 DB는 수직 확장, 애플리케이션 서버는 수평 확장
