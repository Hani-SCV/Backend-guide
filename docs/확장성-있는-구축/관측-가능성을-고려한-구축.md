### 🔍 개념

- **관측 가능성(Observability)**: 시스템 내부 상태를 **외부에서 쉽게 이해하고 모니터링**할 수 있는 능력
- 목표: 문제 발생 시 **빠른 원인 파악 및 대응**, 서비스 안정성 확보
- 구성 요소: **로그(Log) + 지표(Metrics) + 추적(Tracing)**

---


### ⚙️ 핵심 요소

1. **로그(Log)** 📝
    - 시스템 이벤트 기록
    - 예: 사용자 로그인, API 호출, 오류 발생
    - 활용: 문제 재현, 감사(audit), 분석
2. **지표(Metrics)** 📊
    - 수치 기반 성능 모니터링
    - 예: CPU 사용량, 요청 처리 시간, 트래픽 양
    - 활용: 경보(Alert), 트렌드 분석, SLA 모니터링
3. **추적(Tracing)** 🔗
    - 요청 흐름 추적
    - 예: 마이크로서비스 간 API 호출 순서, 지연 시간
    - 활용: 병목 구간 파악, 성능 최적화

---


### ✅ 장점

- 문제 원인 파악 시간 단축 ⏱️
- 시스템 성능 및 상태 가시화 🌐
- SLA/SLO 기반 모니터링 가능 ✅
- 배포 전후 성능 변화 확인 가능 🔄

---


### ⚠️ 단점 / 주의점

- 수집 데이터가 많으면 저장/분석 비용 증가 💰
- 로그/지표 설계 부실 시 의미 없는 데이터 수집
- 관측 도구/인프라 구축 필요 (Prometheus, Grafana, ELK 등)

---


### 🛠 구현 예시


### Prometheus + Grafana (지표 수집 및 시각화)


```yaml
# prometheus.yml
scrape_configs:
  - job_name: 'app'
    static_configs:
      - targets: ['localhost:8080']
```

- Grafana 대시보드 연동 → 실시간 모니터링

### 로그 수집 (ELK Stack 예시)


```plain text
Application → Logstash → Elasticsearch → Kibana
```


### 분산 추적 (Jaeger)


```plain text
서비스 A → 서비스 B 호출 → Trace ID 연결
Kibana/Jaeger에서 요청 흐름 시각화
```


---


### 💡 활용 사례

- 마이크로서비스 환경 문제 추적 🔗
- 트래픽 급증 대응 및 성능 분석 📈
- SLA/SLO 모니터링 및 알람 🚨
- 무중단 배포 후 안정성 검증 🟢
