### 인덱스 (Index)

- 테이블의 특정 컬럼 값을 **빠르게 검색**하기 위한 자료구조
- 키(Key)와 **행 위치(Row Pointer)**를 저장하여 **풀 스캔 방지**
> 인덱스 없으면 WHERE, ORDER BY 등 연산 시 테이블 전체 탐색(Full Scan) 발생

---


### B-Tree & B+Tree 구조


| 항목    | B-Tree              | B+Tree                       |
| ----- | ------------------- | ---------------------------- |
| 구조    | 리프와 내부 노드 모두 데이터 저장 | **리프 노드에만 실제 데이터** 저장        |
| 범위 검색 | 비효율적 (포인터 없음)       | 리프 노드가 **순차 연결**되어 범위 탐색에 유리 |
| 사용    | 이론적 개념              | 대부분 DBMS는 **B+Tree 사용**      |

> 특징: 균형 유지, 탐색/삽입/삭제 = O(log N)

---


### 인덱스 유형


| 종류        | 설명                             |
| --------- | ------------------------------ |
| 단일 인덱스    | 하나의 컬럼 기준                      |
| 복합 인덱스    | 여러 컬럼 조합. **선두 컬럼 조건 중요**      |
| 고유 인덱스    | 중복 값 불허 (Unique Constraint 포함) |
| 비트맵 인덱스   | 값 종류 적은 컬럼에 효율적 (예: 성별)        |
| 전문 검색 인덱스 | 텍스트 내 단어 검색 (Full-Text Index)  |


---


### 인덱스 장단점

- 장점
    - **검색/정렬 속도 향상**
    - **범위 검색 최적화**
    - **JOIN 성능 향상**
- 단점
    - **쓰기 작업 느려짐** (INSERT/UPDATE/DELETE 시 인덱스 재정렬)
    - **디스크 공간 증가**
    - **인덱스 과다 생성 시 성능 저하**

---


### 인덱스 실무 팁

- WHERE절에 자주 사용되는 컬럼 위주 생성
- **카디널리티 높은 컬럼(값 종류 많은)** 우선
- **UPDATE/DELETE 빈번한 테이블**에는 최소화
- 복합 인덱스는 **선두 조건**이 사용되지 않으면 무효
- SELECT * 보단 필요한 컬럼만 조회 (인덱스만으로 결과 조회 가능하면 성능 극대화)
