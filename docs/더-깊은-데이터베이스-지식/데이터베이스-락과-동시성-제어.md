### 개념


> - 동시성 문제를 방지하기 위해 트랜잭션 간의 데이터 충돌을 제어  
>   
> - 목적: 데이터 무결성과 일관성 유지


### 락의 종류

- _공유 락 (S-Lock)_

    → 여러 트랜잭션이 동시에 읽기 가능

- _배타 락 (X-Lock)_

    → 오직 하나의 트랜잭션만 읽기·쓰기 가능

- _의도 락 (Intention Lock)_

    → 테이블 전체 락 전에 페이지·행 단위 락 계획 표시


### 동시성 제어 기법

1. _낙관적 제어 (Optimistic)_
    - 락 없이 수행
    - 커밋 시점에 버전·타임스탬프 충돌 검사
    - 충돌 빈도 낮은 환경에 적합
2. _비관적 제어 (Pessimistic)_
    - 트랜잭션 시작 시 락 획득
    - 충돌 빈도 높은 작업에 적합
3. _2단계 잠금법 (2-Phase Locking)_
    - 확장 단계: 필요한 락 모두 획득
    - 해제 단계: 커밋 후 한꺼번에 해제
    - 직렬화 가능한 격리 보장
4. _MVCC (다중 버전 동시성 제어)_
    - 읽기 시점의 스냅샷 제공
    - 읽기 락 없이 동시 읽기 가능
    - PostgreSQL, InnoDB 등에서 사용

### 주요 이슈 및 해결

- _데드락 (Deadlock)_
    - 서로 상대 락 해제 대기 → 무한 대기
    - 해결: 타임아웃 정책, 락 순서 일관화
- _경합(Lock Contention)_
    - 다수 트랜잭션이 동일 자원 요구 → 성능 저하
    - 해결: 락 범위 최소화, 짧은 트랜잭션 유지
- _락 승격/강등 (Lock Escalation/Demotion)_
    - 너무 많은 행 락 → 테이블 락으로 자동 전환
    - 모니터링 후 인덱스·쿼리 튜닝 필요

### Tip

- 가능한 _행(row)_ 단위 락 사용 (테이블 락 회피)
- 대량 업데이트 시 배치로 나누어 수행
- 짧은 트랜잭션 설계로 락 유지 시간 최소화
- 모니터링 도구(Oracle AWR, PostgreSQL pg_locks) 활용
