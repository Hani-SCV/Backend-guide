### 개념


> - 명령(Command)과 조회(Query)의 **책임을 분리**하는 아키텍처 패턴  
>   
> - 읽기와 쓰기 로직을 **독립적으로 설계**하여 **성능**, **확장성**, **유지보수성** 개선


### 핵심 개념

- **Command**: 상태를 변경 (예: 등록, 수정, 삭제)
- **Query**: 상태를 조회 (예: 목록, 상세 정보 조회)
- **분리 원칙**:
    - 서로 다른 목적 → 각자에 맞는 설계와 인프라 사용 가능
    - 읽기 모델과 쓰기 모델을 따로 구성 (Read/Write 모델 분리)

### 장점

1. 성능 최적화: 읽기/쓰기 작업을 독립적으로 튜닝 가능
2. 확장성 향상: 읽기와 쓰기를 별도로 스케일링 가능
3. 도메인 복잡도 분리: 명령과 조회 모델 분리로 도메인 설계 단순화
4. 비즈니스 로직 분리: 유지보수와 테스트 용이

### 단점

- 구조 복잡도 증가: 모델, API, 저장소가 각각 2세트 필요
- 최종 일관성 문제: 실시간 동기화 아닌 eventual consistency 전제
- 초기 도입 부담: 단순 CRUD에는 과한 설계
