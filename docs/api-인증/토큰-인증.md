### 개념


> - 로그인 후 토큰(예: JWT)을 발급받아 클라이언트가 서버에 요청 시 토큰을 전송해 인증  
>   
> - 서버는 토큰 서명과 만료를 검증해 사용자 신원과 권한을 확인


### 동작 흐름

1. 클라이언트 → 서버: 사용자 자격 증명(아이디·비밀번호) 전송
2. 서버: 자격 증명 확인 후 액세스 토큰, 리프레시 토큰 발급
3. 클라이언트: 이후 요청 헤더에

    ```plain text
    Authorization: Bearer <access 토큰>
    ```

4. 서버: 토큰 서명, 만료, 권한(claim) 검증 → 요청 처리
5. 액세스 토큰 만료 시 리프레시 토큰으로 재발급 요청

### 토큰 갱신 흐름

- 리프레시 토큰 검증 후 새로운 액세스 토큰 발급
- 토큰 회전(token rotation)으로 이전 리프레시 토큰 무효화

```plain text
POST /token/refresh
Content-Type: application/json

{ "refreshToken": "<refresh 토큰>" }
```


### 보안 고려사항

- HTTPS 필수 적용
- 액세스 토큰은 짧은 만료 시간(예: 15분)
- 리프레시 토큰은 httpOnly, secure 쿠키에 저장 또는 안전한 스토리지
- CSRF 방지를 위해 쿠키 대신 Authorization 헤더 사용 권장
- 토큰 탈취 방지: 토큰 블랙리스트, jti(claim) 검증
- 토큰 크기 제한: payload 최소화
