### 개념


> - **복잡하게 만들지 말고, 최대한 단순하게 설계하라**는 원칙  
>   
> - 불필요한 복잡성 제거로 유지보수성과 이해도를 높임


### 핵심 메시지

- 문제 해결 방법은 **가장 간단하고 명확한** 것이 좋다
- 복잡한 설계나 구현은 오류 발생 가능성과 유지보수 비용 증가로 이어짐
- ‘Stupid’는 멸시적 표현이 아니라, 단순함을 강조하는 강한 표현임

### 적용법

- 작은 함수, 적절한 변수명 사용
- 단일 책임 원칙(SRP)과 함께 쓰면 효과적
- 과도한 추상화, 불필요한 패턴 사용 지양
- 코드를 읽는 사람 입장에서 생각하기

### 주의점

- 단순함 추구가 과도한 절차 제거로 이어지면 안 됨
- 단순함과 기능 요구사항 균형 맞추기 필요
