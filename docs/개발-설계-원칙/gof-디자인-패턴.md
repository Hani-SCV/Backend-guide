### 🧠 GOF란?

- **Gang of Four (4인조)**: Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides
- 소프트웨어 설계에서 자주 반복되는 문제들을 해결하기 위한 **23가지 디자인 패턴**을 정리한 책 *"Design Patterns: Elements of Reusable Object-Oriented Software"*에서 유래
- 패턴은 크게 **3가지 범주**로 분류됨:

---


### 📦 생성 패턴 (Creational Patterns)

> 객체 생성 과정을 캡슐화하고, 유연하고 재사용 가능한 객체 생성을 도와줌.

| 패턴 이름                | 설명                      |
| -------------------- | ----------------------- |
| **Singleton**        | 하나의 인스턴스만 생성되도록 보장      |
| **Factory Method**   | 객체 생성을 서브 클래스에 위임       |
| **Abstract Factory** | 관련된 객체 군을 생성하는 인터페이스 제공 |
| **Builder**          | 복합 객체 생성을 단계별로 구성       |
| **Prototype**        | 기존 객체를 복사해서 새 객체 생성     |


---


### 🔗 구조 패턴 (Structural Patterns)

> 클래스나 객체를 조합하여 더 큰 구조를 만드는 방법 제공.

| 패턴 이름         | 설명                    |
| ------------- | --------------------- |
| **Adapter**   | 호환되지 않는 인터페이스를 연결     |
| **Bridge**    | 구현과 추상화 분리            |
| **Composite** | 트리 구조를 통해 객체 그룹 처리    |
| **Decorator** | 객체에 기능을 동적으로 추가       |
| **Facade**    | 복잡한 시스템을 단순 인터페이스로 감쌈 |
| **Flyweight** | 공유 객체를 통해 메모리 절약      |
| **Proxy**     | 대리 객체를 통해 접근 제어       |


---


### 🤹‍♂️ 행위 패턴 (Behavioral Patterns)

> 객체 간의 통신 방법과 책임 분배를 다룸.

| 패턴 이름                       | 설명                            |
| --------------------------- | ----------------------------- |
| **Observer**                | 한 객체 상태 변경 시 관련 객체에 알림        |
| **Strategy**                | 알고리즘을 객체로 캡슐화하여 동적 변경         |
| **Command**                 | 요청을 객체로 캡슐화, 실행 취소 가능         |
| **Template Method**         | 알고리즘의 뼈대를 정의하고 세부는 서브클래스에서 구현 |
| **State**                   | 상태에 따라 다른 행동 수행               |
| **Iterator**                | 컬렉션 요소 순차 접근 제공               |
| **Mediator**                | 객체 간 통신을 중재자에게 위임             |
| **Visitor**                 | 객체 구조 변경 없이 동작 추가             |
| **Interpreter**             | 언어의 문법 표현 및 해석                |
| **Chain of Responsibility** | 요청을 처리할 수 있는 객체를 체인 형태로 연결    |


---


### ✍️ 언제 어떤 패턴을 써야 하나?

- 특정 문제가 반복적으로 발생할 때
- 클래스/객체의 **결합도를 낮추고**, 유연성과 재사용성을 높이고 싶을 때
- **SOLID 원칙**과 함께 쓰면 효과 극대화됨
