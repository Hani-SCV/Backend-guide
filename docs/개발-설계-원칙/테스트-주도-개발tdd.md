### TDD란?


> - **테스트 코드를 먼저 작성**하고, 그 테스트를 통과하는 최소한의 실제 코드를 구현하는 개발 방식  
>   
> - 개발과 테스트를 **동시에 진행**함으로써 코드의 품질과 설계 개선을 유도


### TDD 사이클 (Red → Green → Refactor)

1. **Red**: 실패하는 테스트 작성 (기능이 아직 없음)
2. **Green**: 테스트를 통과하는 최소한의 코드 작성
3. **Refactor**: 코드 리팩토링 (중복 제거, 설계 개선 등)

이 과정을 짧은 주기로 반복


### 장점

- **버그 발생 가능성 감소**
- **기능 요구사항이 명확해짐**
- **리팩토링에 대한 두려움 감소** (테스트가 보호망 역할)
- **작은 단위의 개발 → 빠른 피드백**

### 단점

- 초기 진입 장벽 (생소한 흐름)
- 개발 속도가 느려 보일 수 있음 (단기적으로)
- 단순 CRUD 개발에는 오히려 과도한 느낌이 들 수 있음

### TDD에 어울리는 테스트


| 테스트 종류                 | 설명                        |
| ---------------------- | ------------------------- |
| **단위 테스트 (Unit Test)** | 함수나 메서드 단위의 테스트           |
| **모킹(Mock)**           | 외부 의존성 제거를 위한 가짜 객체 사용    |
| **테스트 커버리지**           | 얼마나 많은 코드가 테스트되고 있는지를 수치화 |


### TDD vs 일반 테스트


| 항목       | TDD    | 일반 테스트  |
| -------- | ------ | ------- |
| 작성 순서    | 테스트 먼저 | 코드 먼저   |
| 피드백      | 빠름     | 느릴 수 있음 |
| 설계 유도    | 강함     | 약함      |
| 테스트 커버리지 | 높음     | 낮을 수 있음 |


### 잘 쓰이는 상황

- 도메인 로직이 복잡할 때
- 장기 유지보수가 예상되는 프로젝트
- 협업 인원이 많은 경우
