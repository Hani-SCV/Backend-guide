### 개념


> - **복잡한 비즈니스 로직**을 효과적으로 모델링하기 위한 설계 방법론  
>   
> - 기술보다 **비즈니스 도메인**을 중심으로 소프트웨어를 설계  
>   
> - **도메인 전문가와 개발자의 협업**을 통해 정확한 모델을 만들고 유지


### 핵심 개념


| 개념                                | 설명                                 |
| --------------------------------- | ---------------------------------- |
| **도메인(Domain)**                   | 비즈니스가 다루는 문제 영역                    |
| **모델(Model)**                     | 도메인을 코드로 표현한 것 (객체/클래스 구조 등)       |
| **유비쿼터스 언어(Ubiquitous Language)** | 도메인 전문가와 개발자가 공유하는 용어              |
| **Bounded Context(경계 컨텍스트)**      | 도메인 모델이 유효한 경계. 각각의 하위 시스템 또는 팀 단위 |


### 주요 구성 요소


| 구성 요소              | 설명                                |
| ------------------ | --------------------------------- |
| **Entity**         | 식별자를 가지며 변경 가능한 객체 (예: 사용자, 주문 등) |
| **Value Object**   | 값 자체로 식별되며 불변 (예: 주소, 금액 등)       |
| **Aggregate**      | 일관성을 보장하는 트랜잭션 단위의 루트 객체          |
| **Aggregate Root** | Aggregate 내에서 외부 접근이 허용된 루트       |
| **Repository**     | Aggregate를 저장/조회하는 인터페이스          |
| **Service**        | 도메인 로직 중 객체에 포함되기 애매한 로직 처리       |
| **Factory**        | 복잡한 객체 생성을 캡슐화                    |


### 아키텍처와 DDD

- DDD는 보통 **계층형 아키텍처**나 **헥사고날 아키텍처 (Hexagonal Architecture)**, **클린 아키텍처**와 함께 사용됨
- 도메인 계층을 중심으로 **비즈니스 로직을 고립**시키는 것이 핵심

### 왜 DDD를 쓰는가?

- 도메인이 복잡한 경우, **요구사항 변화에 유연하게 대응** 가능
- 유지보수가 쉬운 코드 구조 확보
- **협업 생산성 향상** (비즈니스 용어와 코드가 일치)
- 명확한 책임 분리

### 언제 쓰면 안 좋을까?

- 도메인이 단순하거나 CRUD 위주의 시스템
- 도메인 전문가와의 협업이 어렵거나 불명확한 경우
- 개발 리소스가 충분하지 않은 작은 프로젝트
